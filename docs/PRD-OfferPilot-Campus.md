## 1. 产品背景与目标

### 1.1 市场问题

大学生求职面临五大核心痛点，这些痛点在智联招聘官方赛题中被明确指出：

| 痛点 | 现状描述 | 量化影响 |
|------|---------|---------|
| **规划无方向** | 不知道自己适合什么岗位、什么行业，职业规划靠"感觉"和"跟风" | 超过 60% 应届生在求职前没有清晰职业目标 |
| **简历无头绪** | 不知道怎么写简历，一份简历投所有岗位，关键词不匹配 | 70% 简历在 ATS 系统中因关键词不匹配被过滤 |
| **投递无回复** | 海投几十上百份，石沉大海，不知道被拒原因 | 平均每 50 份投递仅获 1-2 次面试机会 |
| **招聘信息噪声大** | 虚假岗位、过期 JD、挂羊头卖狗肉的岗位充斥各平台 | 近 40% 在线岗位存在信息失真 |
| **AI 工具碎片化** | 简历用一个工具、面试用另一个、投递又一个，缺乏串联 | 切换 5+ 工具完成一次完整求职流程 |

### 1.2 产品定义

**OfferPilot Campus** 是一个面向在校大学生的 AI 求职操作系统。它不是"又一个简历修改器"，而是一个将求职全流程串联的决策辅助平台。

**一句话价值主张**：
> 让大学生从"盲目海投"变成"证据驱动的精准求职"——每个岗位都给出可解释匹配、简历改写、招聘官视角反馈和下一步行动建议。

### 1.3 产品目标

#### 业务目标（比赛维度）

| 评审维度 | 初赛权重 | 复赛权重 | 产品对策 |
|---------|---------|---------|---------|
| 创新性 | 25% | 35% | 7 模块闭环 OS，HR 模拟器为独创亮点 |
| 可落地性 | 25% | 20% | MVP 可运行、可演示、可复现 |
| 技术实现 | 20% | 10% | FastAPI + Next.js + LLM，成熟技术栈 |
| 商业价值 | 15% | 20% | B2C 订阅 + B2B2C 高校 + 平台合作 |
| 用户体验 | 15% | 15% | 3 分钟看懂闭环，5 分钟完成端到端演示 |

#### 用户目标

| 指标 | 当前基线 | 目标值 |
|------|---------|--------|
| 单岗位评估时间 | 30+ 分钟人工判断 | 3-5 分钟 AI 辅助 |
| 每份 JD 输出匹配证据数 | 0（凭感觉） | ≥ 5 条可验证证据 |
| 每次投递前准备物 | 0（裸投） | 1 份定制简历 + 1 份风险报告 + 1 组面试题 |
| 求职决策质量 | 盲目海投 | 每个岗位有匹配分、风险、行动建议 |

### 1.4 产品边界

**MVP 范围（本次参赛交付）**：

- 7 个核心功能模块（详见第 3 节）
- Web 应用（浏览器访问）
- 本地优先数据存储
- 手动 JD 输入 + CSV 批量导入
- 模拟数据 + 少量真实 JD 用于演示

**不在 MVP 范围内**：

- 招聘平台自动登录/投递（合规风险）
- 移动端 App
- 多人协作（复赛增强）
- 实时招聘信息聚合爬虫
- 付费/支付系统

### 3.1 模块一：求职画像 Onboarding

**优先级**: P0  
**解决痛点**: 规划无方向  
**模块目标**: 让系统"认识"用户，建立个性化基础

#### 3.1.1 功能列表

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F1.1 | 简历上传与解析 | P0 | 支持 PDF/DOCX/TXT/Markdown 格式上传，使用 MarkItDown 解析为结构化数据（教育背景、技能、项目经历、实习经历、荣誉奖项） |
| F1.2 | 基本信息填写 | P0 | 姓名、学校、专业、年级、毕业年份（可从简历自动提取，用户确认） |
| F1.3 | 求职偏好设置 | P0 | 目标岗位类型（多选）、目标城市（多选）、期望薪资范围、期望行业（多选）、避雷项（如不接受 996、不接受出差等） |
| F1.4 | 技能自评 | P1 | 用户对系统提取的技能进行 1-5 星自评，补充简历中未体现的能力 |
| F1.5 | 画像可视化 | P1 | 雷达图展示技能分布、岗位匹配热力图、求职竞争力评分 |
| F1.6 | 画像编辑 | P0 | 随时可修改已填写信息，修改后触发相关模块重新计算 |

#### 3.1.2 数据模型

```python
class UserProfile(Base):
```

### 3.2 模块二：岗位雷达

**优先级**: P0  
**解决痛点**: 招聘信息噪声大  
**模块目标**: 从海量岗位中筛选出值得关注的，给出质量评分

#### 3.2.1 功能列表

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F2.1 | JD 手动输入 | P0 | 文本框粘贴岗位描述，自动解析为结构化 JD |
| F2.2 | JD 批量导入 | P0 | 支持 CSV/Excel 文件批量导入多个 JD（字段：公司名、岗位名、JD 文本、来源链接） |
| F2.3 | JD 结构化解析 | P0 | 从 JD 文本中提取：公司名称、岗位名称、岗位职责、硬性要求（学历/专业/技能）、软性要求（经验/证书）、薪资范围、工作地点、关键词标签 |
| F2.4 | 岗位质量评分 | P0 | 对每个岗位给出 1-100 质量分，评分维度：信息完整度、薪资透明度、公司可信度、JD 规范度 |
| F2.5 | 岗位去重 | P1 | 检测重复/相似 JD（同公司同岗位），自动合并或提示 |
| F2.6 | 风险提示 | P1 | 对可疑岗位标记风险标签（如：薪资异常高/低、无公司信息、JD 过于模糊、疑似外包） |
| F2.7 | 岗位列表与筛选 | P0 | 表格展示所有岗位，支持按质量分、匹配分、城市、薪资、行业筛选排序 |
| F2.8 | 岗位详情页 | P0 | 展示岗位完整信息、结构化解析结果、质量评分详情 |

#### 3.2.2 数据模型

```python
class JobPost(Base):

    keywords: list[str]              # 关键词标签
```

#### 3.2.3 验收标准

- [ ] 粘贴 JD 文本后 5 秒内完成结构化解析
- [ ] 关键信息提取正确率 ≥ 80%（公司名、岗位名、薪资、城市）
- [ ] 质量评分对明显低质 JD（空壳公司、模糊描述）得分 < 40
- [ ] 批量导入支持 50 条/次
- [ ] 去重检测准确率 ≥ 90%

### 3.3 模块三：匹配解释

**优先级**: P0  
**解决痛点**: 投递无回复、规划无方向  
**模块目标**: 让用户知道"为什么适合/不适合"以及"怎么补"

#### 3.3.1 功能列表

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F3.1 | 匹配评分 | P0 | 对用户简历与目标 JD 进行匹配评分，输出 1-100 总分 |
| F3.2 | 维度评分 | P0 | 分维度评分：教育匹配、技能匹配、经验匹配、项目匹配、软性要求匹配 |
| F3.3 | 证据命中 | P0 | 列出简历中命中 JD 要求的具体证据（如：JD 要求 Python，简历中"使用 Python 完成数据清洗项目"） |
| F3.4 | 缺口分析 | P0 | 列出 JD 要求但简历未覆盖的缺口（如：JD 要求 SQL，简历无相关技能或项目） |
| F3.5 | 机会等级 | P0 | 根据总分给出机会等级：强力推荐（≥80）、值得尝试（60-79）、需要提升（40-59）、暂不建议（<40） |
| F3.6 | 风险提示 | P0 | 列出可能被 HR/ATS 筛掉的风险点（如：学历不匹配、关键技能缺失、经历不相关） |
| F3.7 | 改进建议 | P0 | 针对每个缺口和风险给出具体可执行的改进建议 |
| F3.8 | 匹配对比 | P1 | 多个岗位横向对比，帮助用户选择最优投递目标 |
| F3.9 | 证据引用溯源 | P0 | 每条证据/缺口标注来源：简历第几段、JD 第几条要求 |

- [ ] 每条证据可溯源到简历/JD 具体位置

### 3.4 模块四：简历改写

**优先级**: P0  
**解决痛点**: 简历无头绪  
**模块目标**: 针对特定 JD 一键生成定制简历内容

#### 3.4.1 功能列表

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F4.1 | 简历摘要改写 | P0 | 根据 JD 关键词和用户画像，生成针对性的简历摘要/求职意向 |
| F4.2 | 技能区优化 | P0 | 根据 JD 要求重新排序和补充技能，将匹配技能置顶 |
| F4.3 | 项目经历 STAR 优化 | P0 | 用 STAR 法（Situation-Task-Action-Result）重写项目经历，增加量化结果 |
| F4.4 | 经历裁剪 | P1 | 根据 JD 相关度排序经历，隐藏不相关经历，突出匹配经历 |
| F4.5 | 定制简历预览 | P0 | 生成完整的定制简历 HTML 预览，支持浏览器打印为 PDF |
| F4.6 | 改写对比 | P0 | 左右对比原始简历 vs 定制简历，高亮改动部分 |
| F4.7 | 多版本管理 | P1 | 为同一岗位保存多个简历版本，支持版本回退 |
| F4.8 | 不编造规则 | P0 | 硬约束：不编造用户未提供的经历、技能或数据，所有改写基于原始简历素材 |

#### 3.4.2 改写规则

**必须遵守的红线**：
1. **不编造**：不添加用户简历中不存在的经历、技能或成就
2. **可溯源**：每处改写标注原始素材来源
3. **用户确认**：改写结果展示后由用户确认，不自动替换原始简历
4. **STAR 完整**：项目经历必须包含 Situation、Task、Action、Result 四要素

**改写策略**：
- **摘要**：提取 JD 核心关键词 + 用户核心优势，生成 2-3 句话
- **技能**：JD 要求的技能置顶，注明熟练程度，不夸大
- **项目**：用 STAR 法重写，Action 部分融入 JD 关键词，Result 补充量化数据（基于用户提供的原始数据，不虚构）
- **求职意向**：明确写 JD 中的岗位名称和关键方向

### 5.2 LLM 集成约束

```python
class LLMConfig:
    """LLM 调用统一配置"""
    providers = {
        "openai_compatible": {
            "base_url": "env:OPENAI_BASE_URL",
            "api_key": "env:OPENAI_API_KEY",
            "model": "env:OPENAI_MODEL",
        },
        "aliyun": {
            "base_url": "https://dashscope.aliyuncs.com/compatible-mode/v1",
            "api_key": "env:ALIYUN_API_KEY",
            "model": "qwen-plus",
        },
        "ollama": {
            "base_url": "http://localhost:11434/v1",
            "model": "qwen2.5:7b",
        },
    }
    # 默认优先级: openai_compatible > aliyun > ollama
    # 比赛期间优先使用阿里云/扣子平台官方算力
```

**LLM 使用规范**：
1. 所有 LLM 输出必须经过 Pydantic schema 校验
2. 结构化输出使用 JSON mode 或 function calling
3. 长文本生成使用流式输出 (SSE)
4. 失败重试最多 3 次，回退到下一个 provider
5. 所有 prompt 模板版本化管理

### 5.3 数据存储约束

- **本地优先**：所有用户数据默认存储在本地 SQLite
- **隐私保护**：简历原文不上传到第三方服务（使用 Ollama 时）
- **数据目录**：`~/.offerpilot/` 或项目目录下 `data/`
- **备份导出**：支持导出用户数据为 JSON

## 7. 风险评估

### 7.1 产品风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| LLM 输出质量不稳定 | 高 | 匹配/改写质量波动 | Pydantic schema 校验 + 重试 + 兜底模板 |
| 简历解析准确率不足 | 中 | 结构化数据错误影响后续 | 多格式支持 + 手动编辑兜底 + 解析结果人工确认 |
| 功能范围蔓延 | 高 | 赶不上提交截止 | 严格 P0/P1/P2 优先级，P0 完成前不碰 P1 |
| 演示效果不佳 | 中 | 评委难以在 3-5 分钟理解价值 | 预置演示数据 + 演示脚本 + 关键动画 |
| 评分不可解释 | 中 | 评委质疑可信度 | 每个分数必须有证据链：JD 要求 → 简历证据 → 缺口 → 行动 |

### 7.2 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| LLM API 限流/不可用 | 中 | 演示时卡顿 | 本地 Ollama 兜底 + 缓存常见结果 |
| 文件解析兼容性 | 中 | 部分简历格式解析失败 | MarkItDown + 手动粘贴兜底 |
| 前后端联调延迟 | 中 | 功能无法完整展示 | 先做 API mock，后端先行 |
| SQLite 并发限制 | 低 | 多标签页同时操作冲突 | 个人使用场景，单用户无并发问题 |

### 7.3 合规风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 简历隐私泄露 | 低 | 高 | 本地存储 + 不上传第三方 + 支持 Ollama |
| LLM 编造经历 | 中 | 高 | 硬约束不编造 + 用户确认 + 原文对比 |
| get_jobs 许可证 | 低 | 中 | 不直接复制，独立实现，仅作参考 |
